# 问题复盘 · Problem Retrospect

> 「复盘不是为了追责，是为了不再犯同样的错。」

## 核心理念

一个好复盘必须回答4个问题：
1. **发生了什么？**（事实，不是观点）
2. **为什么发生？**（根因，不是表象）
3. **我们做了什么？**（行动，不是承诺）
4. **下次怎么办？**（机制，不是口号）

**关键区分**：复盘是系统性学习，不是追责大会。目标是改进机制，不是惩罚个人。

---

## 支持的方法论

| 方法论 | 模式名称 | 适用场景 | 时间投入 |
|-------|---------|---------|---------|
| **丹纳赫5Why** | 标准模式 | 复杂问题、向上汇报 | 30-60min |
| **AAR** | 快速模式 | 任务/项目快速回顾 | 15-30min |
| **A3** | 可视化报告 | 系统性问题、跨部门协调 | 1-2h |

---

## 技能类型识别

| 复盘类型 | 特征 | 执行路径 |
|---------|------|---------|
| **事前复盘** | 问题刚暴露，结果未发生 | 预防路径：重点在决策回溯+风险预警 |
| **事后复盘** | 结果已发生，需要总结 | 总结路径：完整5阶段流程 |
| **中期复盘** | 项目进行中，发现偏差 | 调整路径：重点在差距分析+即时纠正 |

## 如何使用

### 模式选择（Phase 0确定）

| 模式 | 触发条件 | 流程差异 |
|------|---------|---------|
| **标准模式** | 默认、复杂问题、需向上汇报 | 完整5阶段+5Why根因 |
| **AAR快速模式** | 时间紧迫、任务回顾、小问题 | Phase 1用AAR 4问，跳过Phase 2 |
| **A3可视化模式** | 跨部门问题、需可视化呈现 | Phase 5输出A3一页纸格式 |

### 全流程（标准模式）

```
触发技能 → Phase 0确定类型和模式 → Phase 1采集事实 → 检查点确认
→ Phase 2分析差距 → Phase 3根因深挖 → 检查点确认
→ Phase 4制定方案 → Phase 5生成报告
```

### AAR快速复盘流程

```
触发技能 → Phase 0选择AAR模式 → AAR 4问快速采集 → 直接根因分析
→ 方案制定 → 简版报告
```

### 单阶段调用

- 「只做根因分析」→ 单独执行Phase 3
- 「帮我整理事实」→ 单独执行Phase 1
- 「生成汇报文档」→ 单独执行Phase 5
- 「用AAR做个快速复盘」→ AAR快速模式
- 「生成A3报告」→ Phase 5输出A3格式

## Phase详情

### Phase 0: 入口分流（复盘主持人）

**Goal**: 确定复盘类型、模式选择和基本信息

**执行内容**：

1. **选择复盘模式**：
   | 模式选择条件 | 推荐模式 |
   |-------------|---------|
   | 时间紧迫（<30min） | AAR快速模式 |
   | 复杂问题、需向上汇报 | 标准模式（丹纳赫5Why） |
   | 跨部门问题、需可视化 | A3可视化模式 |
   | 用户明确指定模式 | 用户指定模式 |

2. **判断复盘类型**：
   - 事前复盘 → 预防路径
   - 事后复盘 → 总结路径
   - 中期复盘 → 调整路径

3. **确认基本信息**：
   - 问题发生的项目/产品名称
   - 关键时间节点
   - 涉及的主要角色/部门
   - 问题的影响范围

4. **判断是否需要追问**：
   | 信息完整度 | 处理方式 |
   |---------|---------|
   | 用户已提供详细文档 | 直接进入Phase 1 |
   | 用户只描述了问题现象 | 启动追问收集信息 |
   | 信息无法追溯 | 标注局限性，基于现有信息分析 |

**追问策略**（信息不完整时）：

按优先级追问，最多3轮：

| 优先级 | 追问维度 | 典型问题 |
|-------|---------|---------|
| P0 | 目标与结果 | 「最终结果是什么？与预期目标差距多大？」 |
| P1 | 关键决策 | 「最关键的承诺/决策是什么时间、谁做的？」 |
| P2 | 协作机制 | 「相关方之间是否有固定同步机制？频率如何？」 |
| P3 | 历史经验 | 「类似问题之前发生过吗？当时怎么处理的？」 |

**Key rule**: 追问不超过3轮，避免变成问卷调查。如果用户已表达清晰，不追问直接推进。

### Phase 1: 事实采集（信息采集员）

**Goal**: 建立客观事实基础，区分事实vs观点

**执行方式**：启动4个并行Agent采集不同维度信息

| Agent | 采集目标 | 输出section |
|-------|---------|------------|
| **时间线Agent** | 关键事件的时间顺序、里程碑节点 | 时间线表格 |
| **决策链Agent** | 关键承诺/决策的背景、依据、执行者 | 决策记录表 |
| **沟通记录Agent** | 正式沟通频率、渠道、关键信息传递 | 沟通分析表 |
| **资源变化Agent** | 资源投入变化、外部约束变化 | 资源变化表 |

**每个Agent的执行模板**：

```
采集任务：[维度名称]

搜索方向：
- [具体信息点1]
- [具体信息点2]
- [具体信息点3]

输出要求：
- 区分「事实」vs「观点」vs「推测」
- 每条信息标注来源（用户输入/文档/推断）
- 发现矛盾时保留矛盾，不要调和
```

**事实vs观点区分表**：

| 类型 | 特征 | 示例 |
|------|------|------|
| **事实** | 可验证、客观发生 | 「4月20日承诺515上线」 |
| **观点** | 主观判断、价值评估 | 「市场认为这是关键功能」 |
| **推测** | 基于信息推断、需标注 | 「推测当时未做技术评估」 |

**Key rule**: 好复盘基于事实，不是观点。观点可以存在但要明确标注。

---

### Phase 1.5: 事实确认检查点

**所有Agent完成后，展示事实摘要**：

```
┌──────────────────┬──────────┬──────────────────────────┐
│ 维度             │ 信息数量  │ 关键发现                  │
├──────────────────┼──────────┼──────────────────────────┤
│ 时间线           │ N个节点   │ 关键转折: ...             │
│ 决策链           │ N个决策   │ 关键承诺: ...             │
│ 沟通记录         │ N次沟通   │ 正式沟通频率: ...         │
│ 资源变化         │ N次变化   │ 主要变化: ...             │
├──────────────────┼──────────┼──────────────────────────┤
│ 事实/观点/推测    │ 统计      │ 事实X条, 观点Y条, 推测Z条 │
│ 矛盾点           │ N处      │ [矛盾描述]                │
│ 信息不足维度      │ [有/无]   │ [缺失维度列表]            │
└──────────────────┴──────────┴──────────────────────────┘

用户确认：继续 / 补充信息 / 返回调整
```

**意义**：事实质量决定了复盘的上限。垃圾进垃圾出，在这里拦截比后期返工成本低。

---

### Phase 2: 差距分析（分析师）

**Goal**: 量化目标与结果的差距，识别差距类型

**丹纳赫差距分析框架**：

| 维度 | 目标 | 实际 | 差距 | 差距类型 |
|------|------|------|------|---------|
| 时间 | [预期时间] | [实际时间] | [差距天数] | 延期/提前 |
| 功能范围 | [预期功能] | [实际功能] | [缺失功能] | 范围缩减/扩大 |
| 质量 | [预期质量] | [实际质量] | [质量差距] | 质量债务 |
| 资源 | [预期投入] | [实际投入] | [额外投入] | 成本超支 |
| 影响 | [预期影响] | [实际影响] | [影响差距] | 影响扩大/缩小 |

**差距类型分类**：

| 差距类型 | 定义 | 复盘重点 |
|---------|------|---------|
| **执行差距** | 目标明确，执行未达标 | 为什么没执行到位？ |
| **认知差距** | 目标设定时就存在偏差 | 为什么目标设定有偏差？ |
| **能力差距** | 能力不足以支撑目标 | 能力短板在哪里？ |
| **环境差距** | 外部环境变化导致偏差 | 环境变化是否可预判？ |

**Key rule**: 差距分析必须量化，不要用「差距很大」这种模糊描述。

---

### Phase 3: 根因深挖（根因分析师）

**Goal**: 用5Why分析法挖到系统性根因

**丹纳赫5Why分析法**：

```
Why 1: 为什么会出现这个差距？
  → [直接原因]

Why 2: 为什么会有这个直接原因？
  → [深层原因]

Why 3: 为什么会有这个深层原因？
  → [机制原因]

Why 4: 为什么机制是这样的？
  → [系统性原因]

Why 5: 为什么系统性原因存在？
  → [根因：组织/流程/认知层面的系统性问题]
```

**根因分类矩阵**：

| 根因类型 | 特征 | 改进方向 |
|---------|------|---------|
| **流程缺失** | 没有标准流程覆盖此场景 | 建立流程 |
| **流程失效** | 有流程但未执行/执行不到位 | 强化执行 |
| **认知偏差** | 对问题重要性/影响判断错误 | 建立认知校准机制 |
| **能力短板** | 团队/个人能力不足以应对 | 能力建设 |
| **协作断层** | 跨部门/角色间信息/责任断层 | 建立协作机制 |
| **环境变化** | 外部约束变化未及时响应 | 建立预警机制 |

**关键决策节点回溯**：

对每个关键决策执行质量评估：

| 决策点 | 决策质量 | 评估依据 | 改进方向 |
|--------|---------|---------|---------|
| [决策1] | 好/中/差 | [评估理由] | [改进建议] |
| [决策2] | 好/中/差 | [评估理由] | [改进建议] |

**Key rule**: 5Why必须挖到系统性原因才算完成。如果停在「某人做错了」，要继续追问「为什么他会做错」。

---

### Phase 3.5: 根因确认检查点

**展示根因链条摘要**：

```
┌─────────────────────────────────────────────────┐
│ 5Why根因链条                                    │
│ Why 1 → Why 2 → Why 3 → Why 4 → Why 5         │
│ [简化的链条描述]                                │
├─────────────────────────────────────────────────┤
│ 系统性根因：[根因类型]                          │
│ 根因描述：[一句话描述]                          │
│ 影响范围：[系统性/局部性]                       │
│ 改进难度：[高/中/低]                            │
├─────────────────────────────────────────────────┤
│ 关键决策评估：                                  │
│ [决策1] 质量: [好/中/差]                        │
│ [决策2] 质量: [好/中/差]                        │
├─────────────────────────────────────────────────┤
│ 用户确认：继续 / 返回深挖                       │
└─────────────────────────────────────────────────┘
```

**意义**：根因分析是主观判断最重的环节，确认后再进入方案制定。

### Phase 4: 方案制定（改进策划师）

**Goal**: 针对根因制定系统性改进方案

**改进方案三维度**：

| 维度 | 方案类型 | 时间框架 | 责任方 |
|------|---------|---------|--------|
| **即时整改** | 针对当前问题的补救措施 | 立即执行 | 具体负责人 |
| **机制优化** | 针对流程/机制的改进 | 中期（1-3月） | 流程负责人 |
| **能力建设** | 针对认知/能力的提升 | 长期（3-12月） | 培训/管理层 |

**方案制定模板**：

每个改进方案需包含：

| 字段 | 内容 |
|------|------|
| **方案名称** | [简洁描述] |
| **针对根因** | [对应的Why层级] |
| **具体措施** | [可执行的动作，不是口号] |
| **责任人** | [具体角色/人名] |
| **完成时间** | [明确日期或周期] |
| **验证方式** | [如何验证方案有效] |

**SOP/检查清单设计**（关键产出）：

```
[场景名称] 检查清单：

□ [检查项1] - [具体检查内容]
□ [检查项2] - [具体检查内容]
□ [检查项3] - [具体检查内容]
□ [检查项4] - [具体检查内容]
□ [检查项5] - [具体检查内容]
```

**Key rule**: 方案必须是「可执行的动作」，不是「以后要注意」这种口号。

### Phase 5: 报告生成（报告撰写员）

**Goal**: 生成适合向上汇报的复盘文档

**报告结构**（向上汇报版）：

```markdown
# [项目/问题名称] 复盘报告

## 一、问题概述与当前状态
- 简洁描述问题和当前状态（2-3句话）

## 二、事实回顾
- 时间线表格
- 沟通记录摘要
- 关键决策记录

## 三、原因分析（5Why根因分析）
- 直接原因
- 深层原因
- 系统性根因

## 四、关键决策节点回溯
- 决策质量评估表

## 五、已采取的行动
- 当前已执行的补救措施

## 六、后续改进措施
- 即时整改表（责任人+时间）
- 机制优化表（责任人+时间）
- SOP/检查清单

## 七、总结与反思
- 核心教训（2-3条）
- 一句话总结

---

*汇报日期：[日期]*
*复盘方法：丹纳赫5Why分析法*
```

**报告语气原则**：

| 原则 | 说明 |
|------|------|
| **客观** | 基于事实，不推诿也不过度自责 |
| **具体** | 用具体事例，不用模糊描述 |
| **行动导向** | 重点在改进措施，不在追责 |
| **系统性** | 强调机制改进，不只谈个人失误 |
| **简洁** | 控制在1500字以内，便于上级阅读 |

**Key rule**: 报告是给上级看的，不是给自己看的。要站在上级视角思考：他想知道什么？他关心什么？

## 关键规则（CRITICAL）

### 必须执行

- **事实优先**：所有分析必须基于事实，观点要明确标注
- **5Why必须挖到底**：不能停在表象或个人失误层面
- **方案必须可执行**：不能用「以后要注意」这类口号
- **报告必须简洁**：向上汇报控制在1500字以内
- **检查点必须确认**：Phase 1.5和3.5必须展示摘要请用户确认

### 禁止执行

- **禁止追责导向**：复盘目标是改进机制，不是惩罚个人
- **禁止调和矛盾**：发现矛盾要保留，不要强行解释
- **禁止编造信息**：信息不足时标注局限性，不要推测填充
- **禁止模糊描述**：差距必须量化，不能用「差距很大」

---

## 诚实边界

### 本技能做不到的事

- **替代用户回忆**：事实信息需要用户提供或文档记录
- **保证完美根因**：根因分析有主观判断，需用户确认
- **预测改进效果**：改进方案需实际执行验证
- **处理无法追溯的问题**：信息缺失过多时需标注局限性

### 复盘的局限

- 所有复盘都是「事后视角」，可能有幸存者偏差
- 根因分析依赖信息质量，信息不足时根因可能偏差
- 改进方案需要组织配合，个人层面改进效果有限

## 快速参考

### 丹纳赫复盘核心公式

```
目标 → 结果 → 差距 → 5Why根因 → 系统性改进
```

### 追问优先级

```
P0: 目标与结果 → P1: 关键决策 → P2: 协作机制 → P3: 历史经验
```

### 事实vs观点标记

```
[F] = 事实（可验证）
[V] = 观点（主观判断）
[I] = 推测（需标注依据）
```

### 报告字数控制

```
向上汇报版：≤1500字
详细分析版：≤3000字
完整归档版：不限
```

## AAR快速模式详解

> After Action Review（事后回顾），源自美国陆军，快速高效的任务复盘框架

### 适用场景

- 时间紧迫（<30分钟）
- 任务/项目完成后快速回顾
- 小问题、不需要深度根因分析
- 团队日常复盘

### AAR核心框架：4个标准问题

| # | 问题 | 目的 | 输出 |
|---|------|------|------|
| 1 | **我们预期发生什么？** | 建立预期基准 | 预期目标陈述 |
| 2 | **实际发生了什么？** | 建立事实基础 | 实际结果陈述 |
| 3 | **为什么会有差异？** | 分析差距原因 | 差距原因列表 |
| 4 | **我们学到了什么？** | 提取经验教训 | 改进要点列表 |

### AAR执行流程（替代标准模式Phase 1-3）

```
Step 1: 预期什么？
  → 快速记录预期目标和计划

Step 2: 实际什么？
  → 快速记录实际发生的事实

Step 3: 为什么差异？
  → 简化版根因分析（1-2层Why即可）

Step 4: 学到什么？
  → 提取1-3条关键教训
  → 生成1-3条改进行动

Step 5: 快速报告
  → 简版复盘报告（500字以内）
```

### AAR关键原则

| 原则 | 说明 |
|------|------|
| **时间短** | 15-30分钟完成 |
| **非正式** | 无责备氛围，全员平等发言 |
| **立即记录** | 复盘结果立即记录、立即分享 |
| **聚焦学习** | 重点在「学到什么」，不在追责 |

### AAR简版报告格式

```markdown
# AAR快速复盘 - [任务名称]

**复盘时间**：[日期]
**参与人员**：[人员列表]

## 1. 预期发生什么
[一句话描述预期目标]

## 2. 实际发生什么
[一句话描述实际结果]

## 3. 为什么有差异
- [原因1]
- [原因2]

## 4. 学到什么
- [教训1] → [改进行动1]
- [教训2] → [改进行动2]

---
*复盘方法：AAR事后回顾*
```

## A3可视化报告模式详解

> 丰田A3报告，一页纸结构化问题解决报告，适合跨部门协调和向上汇报

### 适用场景

- 系统性问题、需要跨部门协调
- 需要可视化呈现复盘结果
- 向上汇报且需要展示完整思考过程
- 需要归档的问题解决记录

### A3报告结构（一页A3纸=297×420mm）

| Section | 占比 | 内容 |
|---------|------|------|
| **标题** | 5% | 问题名称、责任人、日期 |
| **背景** | 10% | 为什么这个问题重要？ |
| **现状** | 15% | 当前情况数据、图表 |
| **目标** | 10% | 期望达成的状态 |
| **根因分析** | 20% | 5Why链条、鱼骨图 |
| **对策** | 15% | 解决方案、实施计划 |
| **效果确认** | 10% | 实施效果、对比数据 |
| **后续行动** | 15% | 标准化、推广、跟进 |

### A3报告执行流程（Phase 5输出格式）

当用户选择A3模式时，Phase 5输出A3格式报告而非标准向上汇报文档：

```
A3报告生成步骤：

Step 1: 整理背景
  → 从Phase 0-1提取背景信息

Step 2: 汇总现状
  → 从Phase 1-2提取事实和差距数据

Step 3: 明确目标
  → 从Phase 4提取改进目标

Step 4: 呈现根因
  → 从Phase 3提取5Why链条
  → 可用鱼骨图可视化

Step 5: 列出对策
  → 从Phase 4提取改进方案

Step 6: 设计效果确认方式
  → 定义如何验证改进效果

Step 7: 规划后续行动
  → 标准化、推广计划

Step 8: 一页纸排版
  → 所有内容控制在A3一页纸内
```

### A3关键原则

| 原则 | 说明 |
|------|------|
| **一页纸** | 强制精炼，所有内容在一页A3纸内 |
| **数据可视化** | 用图表、表格呈现数据 |
| **完整逻辑** | 背景→现状→根因→对策→效果的完整链条 |
| **可执行** | 对策必须有责任人、时间、验证方式 |

