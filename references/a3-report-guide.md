# A3报告编写指南

> 丰田一页纸问题解决报告，适合跨部门协调和向上汇报的可视化复盘格式

---

## 一、A3报告起源与核心理念

### 起源

A3报告源自丰田汽车，是精益制造的核心工具。名称来源于A3纸尺寸（297×420mm），所有问题分析内容强制在一页纸内完成。

### 核心理念

**「一页纸强制精炼」**

A3的核心目的：
- 强制思考精炼，避免冗长报告
- 可视化呈现问题解决过程
- 完整逻辑链条：背景→现状→根因→对策→效果
- 便于跨部门沟通和归档

---

## 二、A3与标准报告的区别

| 特性 | A3报告 | 标准向上汇报 |
|------|--------|-------------|
| **格式** | 一页纸可视化 | 文字文档（1500字） |
| **阅读时间** | 5分钟快速理解 | 10-15分钟阅读 |
| **数据呈现** | 图表、表格优先 | 文字描述为主 |
| **适用场景** | 跨部门协调、归档 | 向上级汇报 |
| **时间投入** | 1-2小时 | 30-60分钟 |

## 三、A3报告结构（标准布局）

### 结构占比

| Section | 占比 | 内容要点 |
|---------|------|---------|
| **标题栏** | 5% | 问题名称、责任人、日期、部门 |
| **背景** | 10% | 为什么重要、影响范围 |
| **现状** | 15% | 数据、图表、问题表现 |
| **目标** | 10% | 期望状态、量化指标 |
| **根因分析** | 20% | 5Why链条、鱼骨图 |
| **对策** | 15% | 解决方案、实施计划表 |
| **效果确认** | 10% | 验证计划、对比指标 |
| **后续行动** | 15% | 标准化、推广、跟进 |

### 结构布局示意

```
┌─────────────────────────────────────────────────────────────┐
│ 标题：[问题名称]  责任人：[姓名]  日期：[日期]  部门：[部门]  │
├───────────────────────┬─────────────────────────────────────┤
│ 背景                  │ 现状                               │
│ （为什么重要）         │ （数据、问题表现）                  │
│                       │                                     │
├───────────────────────┼─────────────────────────────────────┤
│ 目标                  │ 根因分析                           │
│ （期望状态）           │ （5Why链条）                        │
│                       │                                     │
├─────────────────────────────────────────────────────────────┤
│ 对策                                                          │
│ （解决方案表格：对策、责任人、时间、验证方式）                  │
├───────────────────────┬─────────────────────────────────────┤
│ 效果确认              │ 后续行动                           │
│ （验证计划）          │ （标准化、推广）                    │
└───────────────────────┴─────────────────────────────────────┘
```

## 四、各Section编写要点

### 1. 标题栏

**内容**：
- 问题名称（简洁，不超过20字）
- 责任人姓名
- 编写日期
- 所属部门

**要点**：一目了然知道「什么问题、谁负责、什么时候」

### 2. 背景（为什么重要）

**内容**：
- 问题的业务影响
- 为什么需要解决
- 影响范围（用户、业务、团队）

**要点**：2-3句话，让读者理解「为什么要花时间解决这个问题」

**示例**：
```
本问题导致515版本功能交付打折扣，
影响发布会KOL测评体验，可能影响产品口碑和销售。
涉及研发、市场、设备三方协作。
```

### 3. 现状（当前情况）

**内容**：
- 关键数据表格
- 问题具体表现
- 差距量化

**要点**：用数据和图表说话，避免文字堆砌

**数据表格式**：
| 指标 | 当前值 | 目标值 | 差距 |
|------|--------|--------|------|
| 功能完整度 | 60% | 100% | 缺失混剪 |
| 交付时间 | 5月15日 | 5月15日 | 按时但质量打折 |
| 正式沟通次数 | 1次 | ≥3次 | 沟通不足 |

### 4. 目标（期望状态）

**内容**：
- 期望达成的状态
- 量化指标
- 时间节点

**要点**：一句话描述，配合1-2个量化指标

**示例**：
```
目标：建立跨部门协作机制，确保后续项目承诺可兑现。
指标：承诺前技术评审覆盖率100%，跨部门同步频率≥每周1次。
```

### 5. 根因分析（核心）

**内容**：
- 5Why链条
- 可选：鱼骨图

**要点**：必须挖到系统性根因，不能停在个人失误

**5Why链条格式**：
```
问题：功能交付打折扣

Why 1: 承诺时未识别依赖关系
  ↓
Why 2: 评估仓促，缺乏跨部门确认机制
  ↓
Why 3: IPMS新机制，尚未形成标准流程
  ↓
Why 4: 组织变化后缺乏机制宣贯和角色对齐
  ↓
Why 5: 组织变革与流程配套不同步【系统性根因】
```

**鱼骨图分类**（可选）：
| 类别 | 原因 |
|------|------|
| 人员 | 软件PM首次参与IPMS |
| 流程 | 缺乏承诺前置评审流程 |
| 工具 | 无跨部门协作平台 |
| 信息 | 信息传递链路长 |
| 环境 | 发布会时间调整 |

### 6. 对策（解决方案）

**内容**：
- 对策列表
- 责任人
- 完成时间
- 验证方式

**要点**：对策必须对应根因，每条对策有明确责任人

**对策表格式**：
| 序号 | 对策 | 对应根因 | 责任人 | 时间 | 验证方式 |
|------|------|---------|--------|------|---------|
| 1 | 建立承诺前置检查清单 | Why 1-2 | 软件PM | 5月10日 | 检查清单使用记录 |
| 2 | 建立周同步机制 | Why 3-4 | 项目经理 | 5月31日 | 同步会议纪要 |
| 3 | IPMS角色对齐培训 | Why 5 | 培训部门 | 6月15日 | 培训完成证明 |

### 7. 效果确认（验证计划）

**内容**：
- 如何验证对策有效
- 验证时间点
- 对比指标

**要点**：必须有明确的验证方式，不能只写「跟踪效果」

**验证表格式**：
| 对策 | 验证时间 | 验证方式 | 当前值 | 目标值 |
|------|---------|---------|--------|--------|
| 承诺前置检查 | 5月20日 | 检查清单覆盖率 | 0% | 100% |
| 周同步机制 | 6月1日 | 同步会议频率 | 0次/周 | 1次/周 |

### 8. 后续行动（标准化）

**内容**：
- 成功经验标准化
- 推广到其他项目/团队
- 长期跟进机制

**要点**：防止问题再次发生，形成组织资产

**示例**：
```
标准化：
- 承诺前置检查清单纳入项目流程手册
- 跨部门协作SOP写入IPMS指南

推广：
- 组织内其他IPMS项目推广此机制
- 培训材料归档供新项目参考

跟进：
- 每季度回顾协作机制执行情况
- 年度评估机制有效性
```

## 五、A3报告编写流程

| 步骤 | 时间 | 内容 |
|------|------|------|
| 1 | 10min | 整理背景和现状数据 |
| 2 | 5min | 明确目标指标 |
| 3 | 20min | 完成5Why根因分析 |
| 4 | 15min | 设计对策和验证计划 |
| 5 | 10min | 规划后续行动 |
| 6 | 20min | 一页纸排版调整 |
| 总计 | 80min | 约1.5小时 |

---

## 六、A3成功要素

### 必须做到

- ✓ 所有内容在一页A3纸内
- ✓ 数据用图表表格呈现
- ✓ 根因挖到系统性层面
- ✓ 每条对策有责任人、时间、验证方式
- ✓ 后续行动包含标准化

### 禁止做法

- ✗ 内容超出一页纸
- ✗ 大段文字描述（应改用图表）
- ✗ 根因停在「某人失误」
- ✗ 对策没有责任人或时间
- ✗ 只有对策没有效果确认
