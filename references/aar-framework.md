# AAR（After Action Review）事后回顾详解

> 源自美国陆军，快速高效的任务复盘框架，15-30分钟完成

---

## 一、AAR起源与核心理念

### 起源

AAR（After Action Review）由美国陆军开发，最初用于军事训练后的快速回顾。后被广泛应用于企业、医疗、人道主义援助等领域。

### 核心理念

**「学习比追责更重要」**

AAR的核心目的：
- 快速提取经验教训
- 全员平等参与
- 立即记录、立即分享
- 不追求完美分析，追求快速学习

---

## 二、AAR与标准复盘的区别

| 特性 | AAR | 标准复盘（丹纳赫5Why） |
|------|-----|---------------------|
| **时间** | 15-30分钟 | 30-60分钟+ |
| **深度** | 1-2层Why | 5层Why系统性根因 |
| **形式** | 非正式、全员讨论 | 结构化流程 |
| **输出** | 简版报告（500字） | 详细报告（1500字） |
| **适用** | 任务回顾、小问题 | 复杂问题、向上汇报 |

---

## 三、AAR核心框架：4问法

### 标准4问

| # | 问题 | 目的 | 执行要点 |
|---|------|------|---------|
| 1 | **我们预期发生什么？** | 建立预期基准 | 简明陈述预期目标和计划 |
| 2 | **实际发生了什么？** | 建立事实基础 | 只陈述事实，不含观点 |
| 3 | **为什么会有差异？** | 分析差距原因 | 1-2层Why即可，不深挖 |
| 4 | **我们学到了什么？** | 提取经验教训 | 1-3条教训 + 1-3条行动 |

### 问题执行详解

#### 问题1：预期发生什么？

**目标**：建立对比基准

**执行要点**：
- 回顾任务开始时的目标和计划
- 明确预期的关键结果指标
- 不讨论「预期是否合理」，只记录预期

**典型回答**：
- 「预期在5月15日上线完整功能」
- 「预期无重大bug，用户满意度90%」

#### 问题2：实际发生了什么？

**目标**：建立事实基础

**执行要点**：
- 只陈述客观事实，不含主观判断
- 不解释原因，只描述结果
- 用数据说话，避免模糊描述

**典型回答**：
- 「实际5月15日上线，但缺少混剪功能」
- 「实际有3个P1 bug，用户满意度85%」

#### 问题3：为什么会有差异？

**目标**：分析差距原因

**执行要点**：
- 1-2层Why即可，不深挖系统性根因
- 快速列出主要原因（2-5条）
- 区分内部原因和外部原因

**典型回答**：
- 「承诺时未做技术可行性评估」（第1层）
- 「评估时间不足，信息不完整」（第2层）
- 「外部原因：发布会时间调整」

#### 问题4：我们学到了什么？

**目标**：提取经验教训

**执行要点**：
- 每条教训对应一条改进行动
- 行动必须具体可执行
- 不超过3条教训，聚焦关键

**典型回答**：
- 「教训：承诺前需做技术评估」→「行动：建立承诺前置检查清单」
- 「教训：信息不完整时不应承诺」→「行动：信息不足时标注风险」

## 四、AAR执行流程

### 标准流程（15-30分钟）

| 时间 | 步骤 | 内容 |
|------|------|------|
| 2min | 开场 | 说明AAR目的、规则（无责备） |
| 3min | 问题1 | 预期什么？全员快速回答 |
| 5min | 问题2 | 实际什么？全员快速回答 |
| 10min | 问题3 | 为什么差异？讨论主要原因 |
| 5min | 问题4 | 学到什么？提炼教训和行动 |
| 5min | 结束 | 记录结果、确定分享范围 |

### 关键规则

| 规则 | 说明 |
|------|------|
| **全员参与** | 所有相关人员都发言 |
| **平等氛围** | 不分级别，所有人平等发言 |
| **无责备** | 不追责个人，聚焦学习 |
| **不打断** | 发言时不打断、不评判 |
| **立即记录** | 复盘结果立即书面记录 |
| **立即分享** | 记录后立即分享给相关人员 |

## 五、AAR适用场景

### 适合AAR的场景

| 场景 | 原因 |
|------|------|
| 任务/项目完成后 | 快速回顾，提取经验 |
| 敏捷迭代复盘 | 时间短、迭代快 |
| 日常例会复盘 | 定期快速回顾 |
| 小问题分析 | 不需要深度根因 |
| 紧急事件回顾 | 快速总结、立即改进 |

### 不适合AAR的场景

| 场景 | 推荐方法 |
|------|---------|
| 复杂系统性问题 | 丹纳赫5Why |
| 重大质量事故 | 8D、DMAIC |
| 向上级汇报的复盘 | 标准模式+A3报告 |
| 跨部门协作问题 | A3报告 |

## 六、AAR简版报告格式

### 报告模板（500字以内）

```markdown
# AAR快速复盘 - [任务名称]

**复盘时间**：[日期] [时长]
**参与人员**：[人员列表]
**主持人**：[姓名]

---

## 1. 预期发生什么
[一句话描述预期目标和计划]

## 2. 实际发生什么
[一句话描述实际结果和事实]

## 3. 为什么有差异
- [原因1]
- [原因2]
- [原因3]

## 4. 学到什么

| 教训 | 改进行动 | 责任人 |
|------|---------|--------|
| [教训1] | [行动1] | [人名] |
| [教训2] | [行动2] | [人名] |

---

## 分享范围
[哪些人需要看到这份复盘]

---
*复盘方法：AAR事后回顾*
```

## 七、AAR与Retrospect的区别

| 特性 | AAR | Retrospect |
|------|-----|------------|
| **详细程度** | 较简单 | 更详细 |
| **时间跨度** | 单次任务 | 项目阶段 |
| **问题深度** | 表层分析 | 深层分析 |
| **参与范围** | 直接参与者 | 更广泛利益相关方 |
| **后续跟踪** | 较少 | 更正式的跟踪机制 |

---

## 八、AAR成功要素

### 必须做到

- ✓ 时间控制在30分钟内
- ✓ 所有相关人员参与
- ✓ 无责备氛围
- ✓ 立即记录结果
- ✓ 每条教训对应行动

### 禁止做法

- ✗ 追责个人
- ✗ 无限讨论（超出时间）
- ✗ 只记录不分享
- ✗ 教训没有对应行动
- ✗ 领导独断结论
